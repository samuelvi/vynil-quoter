# Secuencia de prompts para generar VinylQuoter con IA

Objetivo: mostrar cómo se podría haber generado este proyecto desde cero usando una IA de forma ordenada: primero comprensión y planificación, después ejecución por fases con Go, Docker de test, Makefile, documentación y verificación.

## Prompt 01 — Contexto y objetivo

```text
Actúa como agente builder. Quiero crear una herramienta CLI llamada VinylQuoter.

Parte de un proyecto vacío: no hay código previo, no hay estructura inicial y no debes asumir que existe ninguna implementación anterior.

Stack obligatorio:
- Lenguaje: Go desde el inicio.
- Ejecución: todos los comandos de aplicación y test deben correr dentro de Docker.
- Docker: configura un entorno de test en `docker/test`.
- Makefile: encapsula los comandos necesarios para construir, levantar, probar, ejecutar y parar el entorno.
- `make run` debe preparar Docker automáticamente si la imagen no existe.
- `make test` debe preparar Docker automáticamente antes de ejecutar los tests.

Objetivo del producto:
- Leer imágenes de portadas de vinilos.
- Identificar a qué álbum corresponden.
- Obtener artista/banda y título del álbum.
- Generar una valoración conservadora en euros.
- Exportar un CSV final con confianza, base de precio y notas.

Antes de implementar:
1. Asume que el repositorio está vacío salvo instrucciones de proyecto.
2. Define la estructura Go inicial (`go.mod`, `cmd/`, `internal/`).
3. Define el Docker de test en `docker/test`.
4. Define los comandos Makefile que encapsulan Docker y Go.
5. Resume objetivo, restricciones y criterios de aceptación.
6. Propón un plan por fases.
7. No escribas código hasta que apruebe el plan.
```

Resultado esperado:
- La IA entiende que el proyecto nace desde cero.
- Define arquitectura Go idiomática.
- Define Docker test y Makefile como base operativa.
- Propone fases pequeñas y verificables.

## Prompt 02 — Diseño funcional

```text
Diseña el flujo funcional de la herramienta.

Debe permitir:
- Procesar una imagen concreta.
- Procesar todas las imágenes de entrada.
- Actualizar el CSV existente sin duplicar imágenes.
- Machacar/regenerar el CSV desde cero.
- Elegir proveedor de reconocimiento: LM Studio local o Gemini.

Menú principal exacto cuando se ejecuta sin flags:
1. Procesar una imagen concreta.
2. Procesar todas las imágenes de `data/src`.
3. `Guardado csv (<ruta actual>)`.
4. `Modelo (<proveedor: modelo actual>)`.
5. Salir.

Comportamiento del menú:
- Debe permanecer abierto después de cada acción.
- Solo debe salir cuando el usuario elige Salir.
- `Guardado csv` debe mostrar la ruta actual entre paréntesis.
- `Guardado csv` debe permitir cambiar ruta CSV, actualizar CSV, regenerar CSV o volver.
- El submenú Guardado csv no debe preguntar por modelo.
- `Modelo` debe mostrar el modelo actual entre paréntesis en el menú principal.
- `Modelo` debe permitir elegir LM Studio por defecto, LM Studio alternativo o Gemini.
- La selección de modelo y ruta CSV debe persistir durante la sesión interactiva.

Entrega:
- Opciones de enfoque con pros y contras.
- Diseño recomendado.
- Estructura inicial de responsabilidades Go.
- Diseño del Docker test en `docker/test`.
- Lista de comandos Makefile propuestos.
- Plan de pruebas.
```

Resultado esperado:
- Menú CLI definido.
- Política update/replace clara.
- Proveedor local por defecto.

## Prompt 03 — Plan técnico por fases

```text
Crea un plan técnico de implementación por fases.

Condiciones:
- Usa TDD siempre que sea posible.
- Cada fase debe tener objetivo, archivos afectados, tests y verificación.
- Separa responsabilidades en módulos pequeños.
- Evita datos reales en Git.
- Go es obligatorio desde la primera fase.
- No propongas prototipos en Python ni migraciones posteriores.
- Docker de test debe vivir en `docker/test`.
- Todos los comandos Go deben ejecutarse dentro del Docker test.
- make run debe preparar Docker automáticamente.
- make test debe preparar Docker automáticamente.
- Añade comandos Makefile para construir, levantar, ejecutar, probar y parar el contenedor.

No implementes todavía. Solo genera el plan.
```

Resultado esperado:
- Plan por fases listo para ejecución.
- Tests definidos antes del código.
- Verificaciones explícitas.

## Prompt 04 — Fase 1: base Go + Docker test + Makefile

```text
Ejecuta la fase 1 del plan.

Desde un repositorio vacío, crea la base del proyecto:
- `go.mod`
- `cmd/vinyl-quoter/main.go`
- paquetes iniciales bajo `internal/`
- Docker test en `docker/test/Dockerfile`
- Compose en `docker/test/docker-compose.yml`
- Makefile con comandos encapsulados.

Comandos mínimos del Makefile:
- make docker-build
- make run
- make test
- make test-build
- make test-up
- make test-down
- make test-shell

Todos los comandos Go deben ejecutarse dentro del contenedor test. `make run` y `make test` deben preparar Docker automáticamente antes de ejecutar Go.

Implementa primero tests que verifiquen:
- Existen los archivos base.
- El Makefile usa Docker para ejecutar `go test ./...`.
- El Compose usa bind mounts explícitos.

Después implementa el mínimo código necesario para pasar los tests.
Reporta comandos ejecutados y resultados.
```

Resultado esperado:
- Proyecto Go inicial.
- Docker test funcional en `docker/test`.
- Makefile como única interfaz operativa.
- Evidencia de RED/GREEN.

## Prompt 05 — Fase 2: CLI y CSV en Go

```text
Ejecuta la fase 2 del plan.

Implementa en Go, primero con tests:
- Selección de una imagen.
- Selección de todas las imágenes soportadas.
- Escritura del CSV.
- Lectura del CSV existente.
- Modo actualizar.
- Modo reemplazar.

Después implementa el mínimo código necesario para pasar los tests.
Reporta comandos ejecutados y resultados.
```

Resultado esperado:
- Comportamiento base cubierto por tests.
- CSV estable.
- Evidencia de RED/GREEN.

## Prompt 06 — Fase 3: proveedores de IA

```text
Ejecuta la fase 3 del plan.

Añade proveedores de reconocimiento:
- LM Studio local compatible con API OpenAI.
- Gemini como fallback.

Investiga los modelos locales disponibles y elige el mejor modelo de visión para imágenes.
Usa ese modelo local por defecto.

No imprimas secretos ni valores de variables de entorno.
Añade tests unitarios sin llamadas reales de red.
Ejecuta los tests mediante Makefile dentro del Docker test.
```

Resultado esperado:
- LM Studio por defecto.
- Gemini opcional.
- Modelo local recomendado documentado.
- Providers testeables sin red.

## Prompt 07 — Fase 4: estructura de datos y Git hygiene

```text
Ejecuta la fase 4 del plan.

Mueve las carpetas de entrada/salida a:
- data/src
- data/dst
- data/report

Ignora el contenido real de data/ en Git, pero conserva las carpetas vacías mediante .gitkeep.

Adapta rutas por defecto, tests, Makefile y documentación.
Verifica que los datos reales se ignoran y los .gitkeep se versionan.
Ejecuta la verificación desde el Docker test mediante Makefile.
```

Resultado esperado:
- Datos fuera del código.
- `.gitignore` correcto.
- Carpetas vacías presentes en Git.

## Prompt 08 — Fase 5: endurecimiento del Docker test

```text
Ejecuta la fase 5 del plan.

Revisa y endurece el Docker de test ya creado en:
- docker/test/Dockerfile
- docker/test/docker-compose.yml

Requisitos:
- Debe servir para ejecutar comandos Go.
- Todos los volúmenes deben ser bind mounts explícitos.
- Las cachés deben vivir bajo .cache/ dentro del proyecto.
- El Makefile debe mantener y documentar:
  - make docker-build
  - make docker-up
  - make docker-down
  - make docker-shell
  - make test-build
  - make test-up
  - make test-down
  - make test-shell
  - make test
  - make run
  - make run-all
- Los comandos de aplicación y test deben depender de `docker-build` para preparar Docker automáticamente.

Verifica Docker Compose config y tests dentro del contenedor.
```

Resultado esperado:
- Entorno reproducible.
- Volúmenes bind explícitos.
- Tests ejecutándose dentro del contenedor.

## Prompt 09 — Fase 6: arquitectura Go completa

```text
Completa la arquitectura Go del proyecto.

Estructura objetivo:
- cmd/vinyl-quoter/main.go
- internal/app
- internal/catalog
- internal/imageinput
- internal/provider
- internal/provider/gemini
- internal/provider/lmstudio
- internal/ui

Condiciones:
- Mantén el menú interactivo y los flags equivalentes.
- El menú interactivo debe permanecer abierto tras cada acción y solo salir con la opción Salir.
- El menú principal debe mostrar `Guardado csv (<ruta actual>)`, modificable desde su submenú.
- El submenú `Guardado csv` no debe preguntar por modelo.
- El menú principal debe mostrar `Modelo (<proveedor: modelo actual>)` y permitir cambiarlo.
- Las acciones posteriores deben reutilizar el CSV y modelo seleccionados en la sesión.
- Mantén LM Studio como proveedor por defecto.
- Mantén Gemini como opción.
- Ejecuta `go test ./...` dentro del Docker test mediante Makefile.
- No añadas implementaciones en otros lenguajes.
```

Resultado esperado:
- Proyecto Go con responsabilidades separadas.
- Tests Go verdes.

## Prompt 10 — Fase 7: documentación

```text
Actualiza la documentación.

README.md debe explicar de forma concisa:
- Qué hace el proyecto.
- Estructura de carpetas.
- Requisitos.
- Comandos Makefile.
- Docker test en `docker/test`.
- Uso directo del CLI.
- Uso mediante Make/Docker sin requerir Go instalado en host.
- CSV generado.
- Política de datos y Git.
- Troubleshooting básico.

Crea además docs/QUICKSTART.md con instrucciones rápidas de uso y enlázalo desde README.md.
```

Resultado esperado:
- README útil sin ser extenso.
- Guía rápida separada.
- Enlace visible desde README.

## Prompt 11 — Fase 8: verificación y cierre

```text
Ejecuta la verificación final antes de afirmar que el trabajo está terminado.

Comandos mínimos:
- docker compose -f docker/test/docker-compose.yml config
- make run --help
- make test
- python3 .opencode/meta/hooks/quality_gate.py --workspace . --mode quick --strict
- git status --short

Si todo pasa, crea un commit Conventional Commit adecuado.
Resume cambios, evidencia y riesgos restantes.
```

Resultado esperado:
- Evidencia antes de declarar finalización.
- Quality gate verde.
- Commit claro.
